# 常见错误与避免方法

## 错误 1：将会话产出直接写入 wiki/

**表现**：用户提问或讨论时，直接在 `concepts/`、`practices/` 或 `queries/` 下创建文档

**避免**：会话中的思考先写入 `thinking/` 对应主题子目录。只有经编译流程（有 raw 来源佐证）的内容才能进 wiki/。未经来源核查的会话产出写入 queries/ 属于此类错误。

## 错误 2：queries/ 写入无来源核查的内容

**表现**：把模型凭记忆或推理得出的结论写入 `wiki/queries/`，没有引用 wiki/ 或 raw/ 中的具体来源

**避免**：queries/ 的核心结论必须指向可核查的来源文档。模型自行推断但无来源支撑的内容应留在 `thinking/` 中。

## 错误 3：忘记添加参考依据

**表现**：回答了问题，但没有链接到相关 wiki 页面

**避免**：Q&A 回答模板中必须包含"参考依据"部分

## 错误 4：归档后不更新 INDEX.md

**表现**：创建了 `wiki/queries/` 下的文档，但 INDEX.md 中没有入口

**避免**：使用写入清单，确保 INDEX.md 的"会话沉淀"部分已更新

## 错误 5：thinking/ 内容当作可信来源引用

**表现**：编译 wiki 文章时，把 `thinking/` 中的结论直接作为事实依据，未回 raw/ 核查

**避免**：`thinking/` 内容不作为可信来源。编译时必须回 `raw/` 找佐证，无法佐证的结论不进 wiki。

## 错误 6：thinking/ 子目录编译后未删除

**表现**：主题已编译进 wiki，但 `thinking/` 下的子目录仍然存在

**避免**：编译完成后必须删除对应的 `thinking/<主题>/` 子目录

## 错误 7：Wikilink 格式错误

**表现**：`[[Harness Engineering|Harness 工程]]` 而不是 `[[Harness-Engineering|Harness 工程]]`

**避免**：参考 Wikilink 格式规范

## 错误 8：大文档只读取开头部分

**表现**：学术论文或长篇技术文章只基于开头部分生成简短摘要

**避免**：使用"大文档处理要求"的检查清单

## 总体执行清单

* [ ] **三层隔离确认**：内容是否在正确的层级？raw/（输入）→ thinking/（草稿）→ wiki/（知识）
* [ ] **任务路由确认**：已阅读任务路由，确认当前任务属于正确场景
* [ ] **Raw Check**: `raw/` 目录中是否包含待处理的新素材（图片/文档）？
* [ ] **Asset Sync**: `raw/images/` 下的所有图片是否已同步到 `wiki/assets/`？
* [ ] **Glossary Lock**: 是否已锁定全局术语表，确保翻译不漂移？
* [ ] **Multimodal Sync**: 图片是否已转化为可编辑的文字解析/Mermaid？
* [ ] **文件名规范**: 所有 wiki 页面文件是否使用 kebab-case（连字符分隔）命名？
* [ ] **Wikilink 格式检查**: 所有 `[[文件名|显示文本]]` 链接中的文件名部分是否与实际文件名完全匹配？
* [ ] **Wikilink Check**: 所有的核心概念是否都已变成 `[[可点击的链接]]`？
* [ ] **Marp Check**: 是否为需要汇报的内容生成了幻灯片格式？
* [ ] **Orphan Check**: 是否存在无法从 `INDEX.md` 触达的"孤儿页面"？

## Linting 检查清单

- [ ] **一致性检查**：扫描 `wiki/`，发现术语冲突时自动统一
- [ ] **孤岛扫描**：识别没有任何链接指向的页面，强制挂载到导航树
- [ ] **queries/ 来源核查**：扫描 `wiki/queries/`，发现无来源引用的文档时报警
- [ ] **thinking/ 停留检查**：扫描 `thinking/` 下存在超过 30 天的子目录，提示编译或清理
- [ ] **thinking/ 编译残留**：检查已编译进 wiki 的主题是否仍保留在 thinking/ 中
- [ ] **补丁发布**：当 `raw/` 有新版本时，在对应 Wiki 页面顶部发布摘要

## 最佳实践提示

* **三层隔离**：raw/ 是输入，thinking/ 是草稿，wiki/ 是知识。不要跨层直接写入
* **手离开键盘**：不要手动修改 `wiki/` 目录下的内容，所有的修改应通过编译流程完成
* **搜索即创作**：把每一次对知识库的提问看作是一次"知识合成"，有价值且经核查的结论存入 queries/
* **结构化思考**：在生成任何长篇文档前，先在 `thinking/` 下构建该主题的工作区
* **利用编译日志**：遇到问题时，先查看 `wiki/compile.log` 了解之前的编译过程
