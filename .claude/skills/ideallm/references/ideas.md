# ideas 管理

## 目录结构

```text
ideas/
└── <project>/
    ├── verified/               # 已验证：你确认过的经验
    ├── referenced/             # 引述：他人观点
    ├── hypothesized/           # 猜想：直觉推断
    ├── inspired/               # 灵感：碎片化想法
    └── *.md                    # 未分类
```

每个项目目录下必须包含四个置信度子目录。项目目录名使用 kebab-case。

## 置信度分类

| 子目录 | 含义 | 编译进 projects 时 | 引用方式 |
|--------|------|-------------------|----------|
| `verified/` | 你确认过的经验 | 直接作为依据 | "已验证：……" |
| `referenced/` | 他人观点 | 可引用，需注明来源 | "据 [[引述来源]] 观点：……" |
| `hypothesized/` | 直觉推断 | 可引用，需标明猜想 | "当前猜想认为：……" |
| `inspired/` | 碎片化灵感 | 仅作思路参考 | "探索方向：……" |
| 未分类 | 待归类 | 先询问用户归类，再决定 | 按归类结果处理 |

## 未分类文件归类

### 归类清单

- [ ] **步骤 1**：扫描 `ideas/<project>/` 根目录下的 .md 文件
- [ ] **步骤 2**：读取文件内容，判断其置信度特征
- [ ] **步骤 3**：向用户确认归类（提供四个选项 + 依据）
- [ ] **步骤 4**：将文件移动到对应子目录

### 判断引导

向用户确认时，提供以下判断依据：

| 问题 | 如果是 | 归入 |
|------|--------|------|
| "你在实际中验证过吗？" | 是 | `verified/` |
| "这是别人说的吗？" | 是 | `referenced/` |
| "你有直觉但没依据？" | 是 | `hypothesized/` |
| "这是一个碎片化灵感？" | 是 | `inspired/` |

### Linting 时的归类

- 每次执行 Linting 时，扫描所有 ideas/ 项目目录下的未分类文件
- 发现未分类文件时，向用户提出归类引导
- 用户也可以主动触发归类

## 项目 Q&A

### 检索范围

`wiki/` + `ideas/`，不含 `thinking/`

### 回答格式

回答后提供"参考依据"，区分来源层级：

```markdown
## 参考依据

### 知识库
- [[Wiki-Article|wiki 文章]]

### 已验证经验
- [[verified/notes|已验证笔记]]

### 引述观点
- [[referenced/notes|引述笔记]]

### 猜想
- [[hypothesized/notes|猜想笔记]]
```

### 引用规则

- wiki/ 内容可作为事实依据
- ideas/verified/ 内容可作为个人经验依据
- ideas/referenced/ 内容需注明来源
- ideas/hypothesized/ 内容需标明猜想性质
- ideas/inspired/ 内容仅作方向参考，不作为结论依据

### Q&A 沉淀

Q&A 中产生有价值的结论时，按置信度写入 `ideas/<project>/` 对应子目录：

| 结论特征 | 写入目录 | 示例 |
|---------|---------|------|
| 有 wiki/ 佐证支撑 | `verified/` | Q&A 中基于 wiki 事实得出的确认性结论 |
| 引用他人观点 | `referenced/` | Q&A 中转述的外部观点 |
| 合理推断但无直接佐证 | `hypothesized/` | Q&A 中的逻辑推演结论 |
| 碎片化灵感 | `inspired/` | Q&A 中萌发的方向性想法 |

写入后，这些结论会参与后续的 projects 编译（触发 ideas/ 哈希变化）。

## ideas/ 文件更新

ideas/ 中的文件是动态的，用户会持续修改。当 ideas/ 文件内容变更时：
- 相关的 projects/ 文章不自动重编译
- Linting 时检测 ideas/ 文件的哈希变化，提示哪些 projects 文章可能需要更新
